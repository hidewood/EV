# 智能充电桩调度计费系统人工验收测试方案

本文档用于课程验收前的人工测试和课堂演示。建议先运行自动化验收脚本确认业务逻辑通过，再按本文档进行页面和流程演示。

## 一、测试前准备

### 1. 启动系统

在项目根目录执行：

```powershell
.\venv\Scripts\python.exe run.py
```

浏览器打开：

```text
http://127.0.0.1:5000
```

进入：

- 用户客户端
- 管理控制台

### 2. 建议先运行自动化测试

```powershell
.\venv\Scripts\python.exe tests\acceptance_course_requirements.py
```

预期：

```text
SUMMARY: 17/17 passed
```

再运行原有单元测试：

```powershell
.\venv\Scripts\python.exe -m unittest tests.test_security_and_queue -v
```

预期：

```text
OK
```

### 3. PowerShell 执行 SQL 脚本方式

PowerShell 不能使用 `< xxx.sql` 这种重定向方式。请使用：

```powershell
mysql -u root -p charging_system -e "source docs/demo-sql/02_reset_small_capacity_manual_demo.sql"
```

或：

```powershell
mysql -u root -p charging_system -e "source docs/demo-sql/01_prepare_batch_dispatch_demo.sql"
```

如果 root 没有密码，提示 `Enter password:` 时直接回车。

## 二、基础人工测试：普通调度、排队、修改、取消、计费

### 1. 管理员设置小容量测试环境

进入管理控制台，设置系统参数：

```text
快充桩数 = 1
慢充桩数 = 1
等候区容量 N = 1
桩队列长度 M = 1
故障调度策略 = 优先级调度
扩展调度模式 = 普通调度
```

保存后，建议重启后端一次，确保运行时配置生效。

然后执行重置脚本：

```powershell
mysql -u root -p charging_system -e "source docs/demo-sql/02_reset_small_capacity_manual_demo.sql"
```

管理员端刷新页面，预期：

```text
只有 #1 快充桩、#3 慢充桩为可用
其它桩为关闭或不参与当前演示
#1 队列 0/1
#3 队列 0/1
等候区为空，容量为 1
```

### 2. 用户注册与登录

在用户端注册 3 个普通用户，例如：

```text
用户A：京A·A0001
用户B：京A·A0002
用户C：京A·A0003
密码：pw 或自定义密码
电池容量：60
```

预期：

```text
注册成功
登录成功
页面进入用户充电申请界面
```

### 3. 快充排队号与调度

用户 A 提交：

```text
充电模式：快充
请求电量：10 度
```

预期：

```text
获得 F 开头排队号，例如 F1
因为快充桩 #1 队列有空位，用户 A 被调度到 #1
管理员端 #1 队列显示 1/1
```

### 4. 等候区排队

用户 B 提交：

```text
充电模式：快充
请求电量：10 度
```

预期：

```text
获得 F 开头排队号，例如 F2
因为 #1 队列已满，用户 B 留在等候区
用户端可看到自己的排队号码
用户端可看到本模式前车等待数量
管理员端等候区显示用户 B
```

### 5. 等候区容量限制

用户 C 提交：

```text
充电模式：快充
请求电量：10 度
```

预期：

```text
提交失败
提示等候区容量不足或类似错误
```

说明：当前快充桩队列满，等候区容量 N=1 且已有用户 B，因此用户 C 不能进入。

### 6. 修改请求电量

用户 B 在等候区修改请求电量：

```text
原请求电量：10 度
修改为：20 度
```

预期：

```text
修改成功
排队号仍然是 F2，不发生变化
管理员端等候区中用户 B 的请求电量更新为 20 度
```

### 7. 修改充电模式

用户 B 在等候区切换模式：

```text
快充 -> 慢充
```

预期：

```text
修改成功
重新生成 T 开头排队号
用户 B 从快充等待队列切换到慢充等待队列
如果慢充桩 #3 有空位，系统可自动调度到 #3
```

### 8. 充电区不允许修改请求

用户 A 在用户端点击开始充电。

然后尝试修改：

```text
修改电量
修改模式
```

预期：

```text
修改失败
提示不在等候区或当前状态不允许修改
```

### 9. 结束充电与详单

用户 A 点击结束充电。

预期：

```text
生成账单/详单
用户端账单记录中出现新账单
点击账单可查看详单
```

详单至少包含：

```text
详单编号
详单生成时间
充电桩编号
充电电量
充电时长
启动时间
停止时间
充电费用
服务费用
总费用
```

### 10. 取消请求

分两种情况测试：

```text
等候区用户取消：不生成详单
充电区用户取消：若已经开始充电，则生成详单
```

预期：

```text
等候区取消后，该用户从等候区消失
充电区取消后，桩队列释放，并生成本次已充电部分的详单
系统自动尝试调度后续车辆
```


## 四、最短完成时间调度测试

### 测试目标

验证系统不是简单地按桩编号分配，而是选择：

```text
等待时间 + 自己充电时间
```

最短的同模式充电桩。

### 操作建议

管理员设置：

```text
快充桩数 = 2
慢充桩数 = 1
等候区容量 N = 10
桩队列长度 M = 2 或 3
扩展调度模式 = 普通调度
```

操作：

1. 用户 A 提交快充较大电量，例如 60 度，进入 #1。
2. 用户 A 开始充电。
3. 用户 B 提交快充较小电量，例如 1 度。

预期：

```text
如果 #2 空闲，用户 B 应优先被调度到 #2
```

可进一步测试：

1. 等用户 A 已充一段时间后。
2. 用户 C 提交更小电量。

预期：

```text
系统会根据 #1 剩余等待时间和 #2 等待时间重新判断最短完成时间
```

## 五、故障调度测试

### 1. 优先级调度

管理员设置：

```text
快充桩数 = 2
慢充桩数 = 1
桩队列长度 M = 1
故障调度策略 = 优先级调度
扩展调度模式 = 普通调度
```

操作：

1. 用户 A 提交快充，进入快充桩 #1。
2. 用户 B 提交快充，进入快充桩 #2。
3. 用户 C 提交快充，留在等候区。
4. 用户 A 开始充电。
5. 管理员对 #1 上报故障。

预期：

```text
#1 状态变为故障
用户 A 当前充电中断并生成详单
故障队列车辆优先进入 pending_reschedule 或等待重调度
等候区普通叫号暂停，用户 C 不应抢在故障队列车辆前面
故障记录中出现 active 状态
```

当 #2 释放空位后：

```text
故障队列车辆优先被调度到 #2
随后系统恢复等候区叫号
```

### 2. 时间顺序调度

管理员设置：

```text
故障调度策略 = 时间顺序调度
快充桩数 = 2
桩队列长度 M = 2 或 3
```

操作：

1. 创建多个快充用户，形成 #1、#2 桩内排队。
2. 对其中一个快充桩上报故障。

预期：

```text
故障桩未充电车辆与其它同类型桩中尚未充电车辆合并
按排队号 F1、F2、F3... 的顺序重新调度
调度期间等候区叫号暂停
调度完成后恢复等候区叫号
```

### 3. 故障恢复调度

操作：

1. 让某个快充桩处于故障状态。
2. 其它同类型快充桩中存在尚未充电车辆。
3. 管理员点击故障恢复。

预期：

```text
恢复桩状态变为可用
系统暂停等候区叫号
同类型其它桩中未充电车辆按排队号重新调度
恢复桩也参与重新分配
调度完成后恢复等候区叫号
```

## 六、扩展调度测试

### 1. 单次调度总充电时长最短

管理员设置：

```text
快充桩数 = 2
慢充桩数 = 1
等候区容量 N = 10
桩队列长度 M = 1
扩展调度模式 = 单次总时长最短
```

操作：

1. 保证多个快充桩同时有空位。
2. 提交多个快充请求，例如一个大电量、一个小电量。

预期：

```text
系统一次可叫多个号
仍按快充请求分配到快充桩
分配结果满足本次进入充电区车辆总完成时长较短
```

### 2. 批量调度总充电时长最短

管理员设置：

```text
快充桩数 = 1
慢充桩数 = 1
等候区容量 N = 1
桩队列长度 M = 2
扩展调度模式 = 批量总时长最短
```

保存后建议重启后端一次。

执行批量调度预置脚本：

```powershell
mysql -u root -p charging_system -e "source docs/demo-sql/01_prepare_batch_dispatch_demo.sql"
```

脚本执行后，管理员端预期：

```text
#1 快充桩：0/2
#3 慢充桩：0/2
等候区有 4 辆 BATCH_DEMO_* 演示车
```

再用一个普通用户账号提交一次充电请求。

预期：

```text
全站车辆数达到 N + 快充桩M + 慢充桩M = 5
触发批量调度
车辆可被分配到任意类型充电桩，不再严格区分快充/慢充偏好
每个桩队列不超过 M，即不会出现 4/2
正确状态应类似：
#1 队列最多 2/2
#3 队列最多 2/2
等候区剩余 1 辆
```

如果出现 `4/2`，说明后端没有重启或仍在运行旧代码。请重启后端并重新执行脚本。

## 七、边界与异常测试

本节用于补充正常流程之外的边界条件。建议在基础流程通过后再执行。

### 1. 等候区容量边界

管理员设置：

```text
快充桩数 = 1
慢充桩数 = 1
等候区容量 N = 1
桩队列长度 M = 1
扩展调度模式 = 普通调度
```

操作：

1. 用户 A 提交快充，占满快充桩队列。
2. 用户 B 提交快充，进入等候区。
3. 用户 C 再提交快充。

预期：

```text
用户 C 被拒绝
系统提示等候区容量不足或请求失败
等待区车辆数不超过 N
桩队列车辆数不超过 M
```

边界点：

```text
当等候区车辆数 = N 时，不能再进入等候区
当同模式桩队列有空位时，可以直接调度，不应错误地受等候区满影响
```

### 2. 桩队列长度 M 边界

管理员设置：

```text
快充桩数 = 1
慢充桩数 = 1
桩队列长度 M = 1
```

操作：

1. 提交第 1 个快充请求。
2. 提交第 2 个快充请求。

预期：

```text
第 1 个请求进入快充桩队列
第 2 个请求不能进入已满桩队列，只能留在等候区
管理员端不会出现 2/1、3/1 等超过 M 的队列显示
```

批量调度模式下也要检查：

```text
每个桩队列不超过 M
如果总车辆数超过所有桩队列空位，剩余车辆应留在等候区
不应出现 4/2 这种超载状态
```

### 3. 排队号边界

操作：

1. 连续提交多个快充请求。
2. 取消其中一个请求。
3. 再提交新的快充请求。

预期：

```text
排队号继续递增
不会因为取消请求而复用已有排队号
快充 F 和慢充 T 独立编号
```

例如：

```text
已有 F1、F2
取消 F2 后再提交快充，应生成 F3，而不是重新使用 F2
```

### 4. 计费时间边界

建议通过自动化验收脚本优先验证：

```powershell
.\venv\Scripts\python.exe tests\acceptance_course_requirements.py
```

人工核对时重点检查以下边界：

```text
07:00：谷时 -> 平时
10:00：平时 -> 峰时
15:00：峰时 -> 平时
18:00：平时 -> 峰时
21:00：峰时 -> 平时
23:00：平时 -> 谷时
跨 23:00 到次日 07:00
```

预期：

```text
正好落在边界时，按新区间计价
跨多个价格区间时，分段计算充电费
服务费始终为 0.8 元/度 * 实际充电度数
总费用 = 充电费 + 服务费
充电时长 = 实际充电度数 / 充电功率
```

示例：

```text
快充 30 度，功率 30 度/小时，09:30 开始，充 1 小时
09:30~10:00 平时 0.5 小时：15 度 * 0.7 = 10.5
10:00~10:30 峰时 0.5 小时：15 度 * 1.0 = 15
充电费 = 25.5
服务费 = 30 * 0.8 = 24
总费用 = 49.5
```

### 5. 请求参数边界

用户端提交请求时测试：

```text
请求电量 = 0
请求电量 < 0
请求电量为空
充电模式为空或非法
重复提交请求
```

预期：

```text
非法电量被拒绝
非法模式被拒绝
同一用户已有活动请求时，不能重复提交新的请求
```

### 6. 修改请求状态边界

分别在以下状态尝试修改电量和模式：

```text
等候区 queuing
已调度 dispatched
充电中 charging
故障重调度 pending_reschedule
已完成 completed
已取消 cancelled
```

预期：

```text
只有等候区 queuing 状态允许修改电量和模式
修改电量不改变排队号
修改模式重新生成对应模式排队号
充电区和其它状态下修改失败
```

### 7. 取消请求状态边界

分别测试：

```text
等候区取消
已调度但未开始充电时取消
正在充电时取消
已完成后再次取消
无活动请求时取消
```

预期：

```text
等候区取消成功，不生成详单
已调度未充电取消成功，不生成详单
正在充电取消成功，生成已充电部分详单
已完成后不再作为活动请求处理
无活动请求时返回无活动请求提示
```

### 8. 管理员参数边界

管理员端修改系统参数时测试：

```text
快充桩数 = 0
慢充桩数 = 0
等候区容量 N = 0
桩队列长度 M = 0
桩队列长度小于当前已有排队车辆数
减少正在使用或已有队列车辆的充电桩数量
非法故障调度策略
非法扩展调度模式
```

预期：

```text
0 或非法参数被拒绝
不能把 M 改到小于当前队列车辆数
不能关闭或减少仍有车辆/会话的充电桩
非法策略被拒绝
```

### 9. 充电桩启停边界

测试：

```text
关闭已有排队车辆的充电桩
关闭正在充电的充电桩
启动已经可用的充电桩
投入运行非待机状态的充电桩
```

预期：

```text
有队列车辆或活跃会话时不允许关闭
状态不匹配的启停操作被拒绝
状态流转应为 off -> standby -> available
```

### 10. 故障边界

测试：

```text
对不存在的充电桩上报故障
对已故障充电桩重复上报故障
恢复非故障状态的充电桩
单个故障桩存在正在充电车辆
单个故障桩只有排队未充电车辆
```

预期：

```text
不存在的桩返回错误
重复故障上报被拒绝
```
