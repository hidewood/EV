# 联合验收交叉联调问题与方案

本文档用于 **EChargeDispatch（我方）** 与 **EV 项目（hidewood/EV，对方）** 联合验收时的交叉联调：我方前端调对方后端、对方前端调我方后端。

**对接原则**：双方直接调用常规业务接口（`/api/*`），**不使用** `/api/v1/external` 外部对接层。用户端、管理员端接口即服务间调用接口。

相关文档：

- [API接口文档.md](./API接口文档.md) — 我方接口规范
- [作业验收用例.md](./作业验收用例.md) — 验收场景
- [联合验收对接开发标准.md](./联合验收对接开发标准.md) — 通用接口约定（可忽略其中 external 章节）

## 1. 联调架构

### 1.1 两种交叉场景

| 场景 | 前端 | 后端 | 说明 |
| --- | --- | --- | --- |
| **A** | EChargeDispatch（Vite `:5173`） | EV（Flask `:8080`） | 改 Vite proxy 指向 EV |
| **B** | EV（静态 HTML） | EChargeDispatch（Django `:8080`） | 改 EV 前端 `API` 常量 + CORS |

两个后端默认均监听 **8080**，**不能同时启动**。联调时按场景切换，或一方改端口（例如 EV 改 8081）。

### 1.2 场景 A：我方前端 → 对方后端

```text
浏览器 :5173
    ↓  /api/*
Vite proxy → http://127.0.0.1:8080
    ↓
EV Flask 后端
```

**配置**（`frontend/vite.config.js`）：

```javascript
proxy: {
  '/api': 'http://127.0.0.1:8080',  // 指向 EV；联调我方后端时改回同一地址即可
}
```

**启动**：

```powershell
# 终端 1：EV 后端
cd e:\code\EV
python -B run.py

# 终端 2：我方前端
cd e:\code\EChargeDispatch\frontend
npm run dev
```

### 1.3 场景 B：对方前端 → 我方后端

```text
浏览器 打开 EV 前端 HTML（静态服务 :8090）
    ↓  跨域 fetch/axios
http://127.0.0.1:8080/api/*
    ↓
EChargeDispatch Django 后端
```

**配置**：

1. EV 用户端 / 管理端 HTML 中，将 `const API = '/api'` 改为：

   ```javascript
   const API = 'http://127.0.0.1:8080/api'
   ```

2. 我方 `backend/config/settings/dev.py` 增加 EV 前端来源：

   ```python
   CORS_ALLOWED_ORIGINS = [
       'http://localhost:5173',
       'http://localhost:8090',
       'http://127.0.0.1:8090',
   ]
   ```

**启动**：

```powershell
# 终端 1：我方后端
cd e:\code\EChargeDispatch\backend
python manage.py runserver 8080

# 终端 2：静态服务 EV 前端（改完 API 地址后）
cd e:\code\EV\frontend
npx serve -p 8090
```

浏览器访问：

- 用户端：`http://localhost:8090/user-client/index.html`
- 管理端：`http://localhost:8090/admin-client/index.html`

## 2. 问题总览

按严重程度排列，**P0 不修则基本流程跑不通**，**P1 影响部分页面**，**P2 仅影响辅助功能**。

| 优先级 | 场景 | 问题摘要 | 建议修改方 |
| --- | --- | --- | --- |
| P0 | A | 无请求时 `queue-status` / `charging/status` 返回 3002 | EV 后端 |
| P0 | A | 开始充电读 `position.pile_id`，EV 不提供 | EV 后端 或 我方前端 |
| P0 | A | 注册不返回 `access_token` | EV 后端 |
| P0 | A | 注册 `user_name` 必填，我方前端可选 | EV 后端 或 我方前端 |
| P0 | B | 无请求时返回 `{ has_request: false }`，EV 前端误判为有请求 | EV 前端 或 我方后端 |
| P1 | A | 充电进度字段 `charged_amount` vs `estimated_amount` | EV 后端 或 我方前端 |
| P1 | A | 桩队列详情响应结构不一致（对象 vs 数组） | EV 后端 或 我方前端 |
| P1 | A | 等候区满：我方 `accepted: false`，EV 错误码 3005 | EV 后端 或 我方前端 |
| P1 | B | 报表字段 `items` vs `piles` | 我方后端 或 EV 前端 |
| P1 | B | 缺少 `/admin/waiting-area` | 我方后端 |
| P2 | A | 缺少 `/admin/acceptance/snapshot` | 验收时用我方后端 |
| P2 | B | 缺少 `/admin/reset/default` | 我方后端 或 EV 前端去掉按钮 |
| P2 | A/B | `ahead_count` 字段位置不一致 | 统一约定后改一方 |

## 3. 场景 A 详细问题与方案

> 我方前端（Vue + `fetch`）调 EV 后端（Flask）

### 3.1 【P0】无活跃请求时状态查询返回错误码

**现象**：用户登录后首页每 5 秒弹「请求不存在」，无法显示「提交充电请求」表单。

**原因**：

| 接口 | 我方后端 | EV 后端 |
| --- | --- | --- |
| `GET /api/charging/queue-status` | `200` + `{ has_request: false }` | 错误码 `3002` |
| `GET /api/charging/status` | `200` + `{ has_request: false }` | 错误码 `3002`（无 active session） |

我方 `frontend/src/api/request.js` 对 `code !== 0` 一律抛错；`HomeView.vue` 并行请求上述两个接口，无请求时必失败。

**方案（推荐改 EV 后端，与联合标准一致）**：

```python
# EV request_controller.py — queue_status
if err == "no_active_request":
    return success({"has_request": False})

# EV charging_service.py — query_charging_state
if not session:
    req = ChargingRequestDAO.find_active_by_car_id(car_id)
    if not req:
        return {"has_request": False}, None
    # 已调度但未开始充电：返回排队信息（见 3.2）
```

`charging/status` 在 **已调度、未开始** 时也应返回 `200`（可含 `has_request: true, status: dispatched, charged_amount: 0`），而非 3002。

**备选（改我方前端）**：`HomeView.vue` 的 `fetchStatus` 捕获 3002 并当作 `{ has_request: false }` 处理（仅联调用，不建议作为长期方案）。

---

### 3.2 【P0】开始充电找不到 `pile_id`

**现象**：状态为 `dispatched` 时点击「开始充电」，提示「尚未分配到充电桩」。

**原因**：

- 我方 `HomeView.vue`：`queueData.value?.position?.pile_id`
- EV `query_state` 返回：`pile_id` 在顶层，`position` 仅含 `{ position, queue_num }`

**方案（推荐改 EV，对齐我方 `position_info`）**：

```python
# EV queue_service.get_car_position — dispatched/charging 分支
return {
    "position": "充电区",
    "queue_num": car_request.queue_num,
    "pile_id": car_request.pile_id,      # 新增
    "ahead_count": ahead,                  # 新增，见 3.7
}
```

**备选（改我方前端）**：

```javascript
const pileId = queueData.value?.position?.pile_id ?? queueData.value?.pile_id
```

---

### 3.3 【P0】注册后不返回 Token

**现象**：注册成功跳转用户页，后续接口全部 401。

**原因**：

- 我方 `RegisterView.vue` 注册成功后 `saveAuth(data)`，依赖 `access_token`
- EV `auth_controller.register` 仅返回 `{ car_id, user_name }`

**方案（推荐改 EV 后端）**：注册成功后签发 JWT，与 login 一致：

```python
return success({
    "access_token": token,
    "expires_in": 7200,
    "car_id": user.car_id,
    "user_name": user.user_name,
    "role": user.role,
})
```

**备选（改我方前端）**：注册成功后自动调用 `login`。

---

### 3.4 【P0】注册时 `user_name` 必填

**现象**：用户只填车牌和密码时，EV 返回 `1001`。

**原因**：EV 校验 `car_id, user_name, password` 均非空；我方 `user_name` 可选，省略时不传该字段。

**方案（二选一）**：

1. **EV 后端**：`user_name` 缺省时默认 `car_id`（与我方 `AuthService.register` 一致）
2. **我方前端**：注册时始终传 `user_name: car_id`（联调前临时改一行）

---

### 3.5 【P1】充电进度字段名不一致

**现象**：充电中进度条始终为 0。

**原因**：

- 我方 `HomeView.vue`：`chargingData.charged_amount`
- EV `query_charging_state`：`estimated_amount`

**方案（推荐 EV 增加别名）**：

```python
return {
    ...
    "estimated_amount": estimated_amount,
    "charged_amount": estimated_amount,   # 兼容我方前端
    "status": "charging",                 # 可选，便于前端统一
}
```

---

### 3.6 【P1】桩队列详情响应结构不一致

**现象**：管理端桩详情页队列空白。

**原因**：

- 我方 `GET /api/queue/pile/{id}` 返回：`{ pile_id, queue_used, queue_total, queue: [...] }`
- EV 返回：**数组** `[{ car_id, position, ... }, ...]`

**方案（推荐 EV 包一层）**：

```python
return success({
    "pile_id": pile_id,
    "queue_used": len(vehicles),
    "queue_total": pile.queue_len if pile else 5,
    "queue": vehicles,
})
```

**备选**：我方 `PileDetailView.vue` 判断 `Array.isArray(data)` 时包装为 `{ queue: data }`。

---

### 3.7 【P1/P2】`ahead_count` 字段

**现象**：排队页显示「前方 0 辆」或开始充电按钮状态不准。

**原因**：

- 我方 `queue-status` 顶层返回 `ahead_count`
- EV 仅在等候区 `position.front_count` 提供，充电区无此字段

**方案**：EV `query_state` 顶层增加 `ahead_count`（与 `position.ahead_count` 保持一致）。

---

### 3.8 【P1】等候区已满的响应语义

**现象**：站内已满时，我方前端按成功响应处理 `accepted: false`，EV 直接返回 3005 错误。

**原因**：我方 `POST /api/charging/request` 可能返回 `{ accepted: false, reason, message }` 且 `code=0`；EV 返回 `code=3005`。

**方案（二选一）**：

1. **EV 对齐我方**：满员时 `200` + `{ accepted: false, reason: "waiting_area_full", message: "..." }`
2. **我方前端**：捕获 3005 并提示「等候区已满」（改动小，联调可接受）

---

### 3.9 【P2】验收快照页不可用

**现象**：`/admin/acceptance` 请求 `/api/admin/acceptance/snapshot` 失败。

**原因**：该接口为我方独有（配合 `run_acceptance` CLI），EV 未实现。

**方案**：验收填表场景 **仍用我方前后端一体**；交叉联调只验证用户端 + 管理端常规流程，不要求 EV 提供快照接口。

## 4. 场景 B 详细问题与方案

> EV 前端（axios + 内联 Vue）调我方后端（Django）

### 4.1 【P0】无请求时 `has_request: false` 导致界面卡死

**现象**：未提交请求时，EV 用户端不显示「提交充电请求」卡片，或显示空白状态区。

**原因**：

- 我方返回 `200` + `{ has_request: false }`
- EV `loadRequest`：`this.request = r.data.data`，对象 truthy；模板 `v-if="!request"` 为 false

**方案（推荐改 EV 前端）**：

```javascript
// EV user-client/index.html — loadRequest
if (r.data.code === 0) {
  const data = r.data.data
  if (data && data.has_request === false) {
    this.request = null
  } else {
    this.request = data
  }
  ...
}
```

**备选（不推荐）**：我方改为无请求时返回 3002，会破坏我方自身前端。

---

### 4.2 【P1】充电进度字段

**现象**：EV 用户端充电进度为 0。

**原因**：EV 读 `estimated_amount`，我方返回 `charged_amount`。

**方案（推荐我方后端增加别名）**：

```python
# ChargingSessionService.get_charging_status 返回中增加
'estimated_amount': float(charged_amount),
```

---

### 4.3 【P1】报表数据结构

**现象**：EV 管理端报表列表为空。

**原因**：

- EV：`reportData = r.data.data?.piles`
- 我方：`{ period, date, items: [...] }`

**方案（二选一）**：

1. **我方后端** `ReportService.generate_stats` 返回中增加 `"piles": results`（与 `items` 同内容）
2. **EV 管理端**：`r.data.data?.piles || r.data.data?.items`

---

### 4.4 【P1】缺少等候区接口

**现象**：EV 管理端仪表盘「等候区」统计不更新。

**原因**：EV 调用 `GET /api/admin/waiting-area`，我方未实现。

**方案（推荐我方后端新增）**：

```http
GET /api/admin/waiting-area
Authorization: Bearer <admin-token>
```

响应示例（与 EV 现有实现对齐）：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "capacity": 10,
    "fast_waiting": 2,
    "slow_waiting": 1,
    "fast_queue": [],
    "slow_queue": []
  }
}
```

实现位置建议：`apps/operations/views.py` + `urls.py`，复用 `QueueService` 统计等候区占用。

---

### 4.5 【P2】缺少演示重置接口

**现象**：EV 管理端「恢复演示默认」按钮失败。

**原因**：EV 调用 `POST /api/admin/reset/default`，我方未实现。

**方案（三选一）**：

1. 我方实现该接口（调用 `init_system` 或清空运行时数据）
2. 联调时 EV 管理端隐藏/禁用该按钮
3. 联调前手动 `python manage.py init_system` 重置数据

---

### 4.6 【P2】CORS 未放行 EV 前端来源

**现象**：浏览器控制台 CORS 错误，接口无法访问。

**方案**：见本文 §1.3，在 `dev.py` 加入 EV 静态服务 origin。生产环境通过 Nginx 反代同源访问，无需 CORS。

## 5. 服务间直接调用约定

不使用 `/api/v1/external` 时，双方系统互调 **与用户/管理员客户端相同** 的接口：

### 5.1 认证

| 用途 | 接口 | 说明 |
| --- | --- | --- |
| 代用户操作 | `POST /api/auth/login` | 用车辆 `car_id` + 密码换 user token |
| 管理端操作 | `POST /api/auth/login` | 管理员编号 `A077379` + 密码 |
| 自动建用户 | `POST /api/auth/register` | 车辆不存在时先注册再 login |

### 5.2 用户端常用链路

```text
POST /api/charging/request
GET  /api/charging/queue-status
POST /api/charging/start
GET  /api/charging/status
POST /api/charging/end
GET  /api/bill/list
POST /api/bill/pay/{bill_id}
```

### 5.3 管理端常用链路

```text
GET  /api/pile/status
POST /api/pile/{id}/poweron | /start | /poweroff
GET  /api/queue/pile/{id}
POST /api/admin/fault/report
POST /api/admin/fault/recover
GET  /api/admin/report/stats
GET  /api/admin/system-config
PUT  /api/admin/system-config
```

### 5.4 调用约束

- 请求体 JSON，`Content-Type: application/json`
- 受保护接口带 `Authorization: Bearer <token>`
- 业务成败看响应 `code`，HTTP 状态码均为 200
- 不要自行计算排队号、费用，以被调方返回为准
- 同一车辆同时只能有一个未完成请求（`3001`）

## 6. 推荐修复分工

联调前双方各改一部分，**优先保证 P0**：

| 负责方 | 修改项 |
| --- | --- |
| **EV 后端** | 3.1 空状态语义；3.2 position 含 pile_id；3.3 注册返回 token；3.4 user_name 默认值；3.5 charged_amount 别名；3.6 队列详情结构 |
| **EV 前端** | 4.1 has_request 判断；4.3 报表 items/piles 兼容 |
| **我方后端** | 4.2 estimated_amount 别名；4.3 piles 别名；4.4 waiting-area 接口 |
| **我方前端** | 可选：3.2 pile_id 回退读取；3.8 捕获 3005（若 EV 不改） |

验收填表（`run_acceptance` + `/admin/acceptance/snapshot`）**不纳入交叉联调**，仍用我方完整栈。

## 7. 联调检查清单

### 7.1 场景 A（我方前端 → EV 后端）

- [ ] EV 后端 `:8080` 已启动，数据库已初始化
- [ ] Vite proxy 指向 EV
- [ ] 用户注册 / 登录成功
- [ ] 无请求时首页不报错，显示提交表单
- [ ] 提交充电请求 → 排队 → 开始充电 → 结束充电 → 账单支付
- [ ] 管理员登录，桩上电 / 投运 / 故障 / 恢复
- [ ] 桩队列详情页有数据

### 7.2 场景 B（EV 前端 → 我方后端）

- [ ] Django `:8080` 已启动，`init_system` 已执行
- [ ] EV 前端 `API` 指向我方，`CORS` 已配置
- [ ] 静态服务 `:8090` 可访问 EV HTML
- [ ] 无请求时 EV 用户端显示提交表单
- [ ] 完整用户充电流程可走通
- [ ] EV 管理端报表、等候区（我方实现 waiting-area 后）正常
